# ACP Harness (C#) — Type Generation Strategy (incl. Discriminated Unions)

## Goals

- **Regeneratable models:** it must be possible to delete generated model files, re-run codegen from the ACP release-pinned schema, and have the project compile.
- **Correct discriminated unions:** union-heavy parts of the ACP schema (`oneOf`/`anyOf`) should be represented in C# as *real* union types (preferably inheritance + discriminator-driven deserialization), not as `object`/extension-data bags.
- **Forward compatible:** unknown or future extension variants must not crash deserialization; preserve the raw JSON.
- **Minimal handwritten model code:** handwritten code should be limited to the generator(s) and small runtime helpers.

## Inputs

- Schema is fetched from ACP **GitHub Release assets** and stored under `schema/schema.json` (pinned to a specific protocol version).

## Current state (as of 2026-04-11)

- NJsonSchema (invoked via `tools/Agent.Acp.TypeGen`) can generate most DTOs, but it **does not reliably model union-heavy areas**.
- A known failure mode is placeholder types such as `Content1` being used in generated DTOs (e.g. `ContentChunk.content : Content1`), which prevents end-to-end typed unions.

## Decision: union generation must be handled by a dedicated generator

### Why

- NSwag/NJsonSchema polymorphism works best with OpenAPI discriminator constructs, and does **not** reliably generate correct C# discriminated unions from pure JSON Schema `anyOf`/`oneOf`.
- We attempted “add converters over existing generated models”, but this does not work end-to-end because union references in core DTOs are still generated as placeholder extension objects (`Content1`).

### What we will do

- Introduce a dedicated **union code generator** (`tools/Agent.Acp.UnionGen`) that reads `schema/schema.json` and emits C# code for selected union types and related DTOs.
- Generated unions are represented as:
  - **abstract base type** + **derived types** per discriminator value (schema-aligned names where possible)
  - generated `JsonConverter<TUnion>` which dispatches based on discriminator property
  - generated fallback `Unknown*` derived type that wraps `JsonElement Raw` to preserve unknown/extension payloads

This preserves forward compatibility and avoids runtime failures when new variants appear.

## Compatibility: schema remains compatible with other implementations

- We prefer **not** to rewrite the upstream ACP schema for runtime use.
- If we add any metadata in the future (e.g. discriminator hints), it must be done using JSON Schema-compatible **extension keywords** that unrelated ACP implementations can ignore.

## Practical pipeline (target)

We will converge on one of these approaches:

### Approach A (preferred): split generation responsibilities

- NJsonSchema generates the majority of object/enum DTOs.
- UnionGen generates:
  - union base/derived types
  - and any DTOs that NJsonSchema currently degrades into placeholders for union refs (e.g. `ContentChunk.content` should be `ContentBlock`, not `Content1`).
- The NJsonSchema input schema is preprocessed to **exclude** those defs so the generated code does not conflict.

Pros:
- minimal custom generation surface
- good regen story
- keeps NJsonSchema for the “boring” 80%

Cons:
- requires careful ownership of which `$defs` are generated by which generator

### Approach B: fully custom schema→C# model generator

Pros:
- single source of generated truth

Cons:
- larger implementation scope

### Approach C: add discriminator metadata + OpenAPI conversion for NSwag

Pros:
- potentially less generator code

Cons:
- may still require structural rewriting; risk of drifting from upstream schema

## Open decisions

### SessionUpdate variants that reference ContentChunk

In ACP schema, `SessionUpdate` includes multiple variants that reference the same `$defs.ContentChunk` but differ by discriminator values:
- `user_message_chunk`
- `agent_message_chunk`
- `agent_thought_chunk`

Options:
1) Generate distinct derived types per discriminator value (more expressive)
2) Use one `ContentChunk` type with a discriminator property (simpler)

(Decision pending.)

## Non-goals

- Perfectly strong-typing explicit extension objects where the protocol intentionally allows arbitrary shapes. Those will be represented as `Unknown*` wrappers over `JsonElement`.
